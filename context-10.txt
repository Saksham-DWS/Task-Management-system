Ans : 
Short take: what you have is a basic RBAC + per‑user ACL list. That’s common for small/medium apps, but it isn’t “production‑grade” by default. It’s acceptable as an MVP, but you’d want to harden it before calling it broadly “globally accepted.”

Key risks in the current style:

Hard‑coded super‑admin email and ad‑hoc locks are brittle; production systems usually model super‑admin as a role + policy, not a magic email.
Access lists stored on the user doc are fine early on, but they’re hard to scale/audit; a membership/access collection is more standard.
No audit trail for role/access changes; production systems log who changed what, when, and from/to.
No elevated‑action safeguards (approval, 2FA, or at least “confirm + reason” for role changes).
Frontend gating is not enough; every backend endpoint must enforce the same rules (some do, but it’s easy to miss).
What’s “more accepted” in production:

Explicit RBAC/ABAC policies (often a permission matrix or policy engine like Casbin/OPA).
Centralized authorization checks on every sensitive endpoint.
Audit logs + alerts for role/access changes.
Clear role hierarchy rules and prevention of “last admin” loss (you already started this).
Data model that supports growth (membership tables/collections with indexes, rather than ever‑growing arrays on user records).
